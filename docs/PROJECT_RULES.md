# 项目初始化规则

## 目标

本项目目标是实现一个基于 Spring Boot 的打卡与备忘录管理系统。当前阶段以“先能跑、再补功能、最后完善体验”为准。

## 当前阶段规则

- 当前仓库处于初始化阶段，不要把未实现模块描述成“已完成”。
- 新增代码必须优先保证可以编译，再考虑功能扩展。
- 所有新功能默认放在 `com.codexzy` 包下对应分层目录中。
- 没有明确需求前，不做多模块拆分。
- 没有明确需求前，保持单体 Spring Boot 应用结构。

## 技术规则

- JDK 版本固定为 17。
- Spring Boot 使用 3.2.x。
- 构建工具使用 Maven。
- ORM 使用 MyBatis-Plus。
- 页面模板使用 Thymeleaf。
- 安全框架使用 Spring Security。
- 数据库使用 MySQL 8.x。

## 分层规则

- `controller` 只处理请求参数、响应和页面路由。
- `service` 负责核心业务逻辑。
- `mapper` 负责数据库访问。
- `entity` 与数据库表结构保持一致。
- `dto` 只承载接口或页面需要的传输字段。
- `config` 只放框架配置。
- `util` 只放无状态通用工具。

## 提交规则

- 每次提交只做一类事情，避免把环境修复、功能开发、文档修改混在一起。
- 提交信息使用清晰前缀，例如 `feat:`、`fix:`、`docs:`、`refactor:`。
- 未确认前不要提交 `.idea/`、`.claude/` 和临时文件。
- `uploads` 目录只保留占位文件，不提交真实上传内容。
- `target/` 必须忽略。

## 开发规则

- 新增功能前先补最小目录和配置，再写业务代码。
- 业务实现优先从数据库模型、实体、Mapper 开始。
- 每完成一个模块，至少补一个基础测试或启动验证。
- 配置项优先写入 `application.yml`，不要把账号密码硬编码到 Java 代码中。
- 文件上传路径统一走配置项 `file.upload-dir`。

## 文档规则

- `README.md` 反映真实项目状态。
- 设计方案放在 `docs/` 下。
- 计划文档可以保留，但要和代码进度一致。
- 如果实现和设计文档不一致，优先更新文档说明差异。

## 启动前检查

启动或编译前先确认：

1. `mvn -v` 显示的 Java 版本是否为 17 或更高。
2. MySQL 是否已启动。
3. `codexzy` 数据库是否已创建。
4. `application.yml` 中数据库连接是否正确。

## 下一阶段

初始化完成后，优先进入以下开发顺序：

1. 数据库建表脚本
2. 实体类与 Mapper
3. 安全配置与登录注册
4. 打卡模块
5. 备忘录模块
6. 前端模板与统计页面
